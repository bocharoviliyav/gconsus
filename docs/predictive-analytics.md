# Предиктивная аналитика -- стратегический документ

## 1. Введение

Данный документ описывает стратегию внедрения предиктивной аналитики в платформу GConsus. Цель -- прогнозирование активности разработчиков, раннее обнаружение аномалий и построение рекомендаций для повышения продуктивности команд.

## 2. Текущее состояние

В системе реализован неактивный ML коннектор (`backend/adapter/ml/connector.go`) с интерфейсом `ml.Client`. Коннектор поддерживает:

- Отправку запроса на прогнозирование (`Predict`).
- Проверку доступности ML-сервиса (`HealthCheck`).
- Stub-реализацию при `ML_ENABLED=false`.

Формат запроса:

- `user_id` -- идентификатор пользователя.
- `metric_name` -- имя метрики (commits, prs, reviews, issues).
- `history_days` -- количество дней исторических данных.
- `forecast_days` -- горизонт прогноза.
- `as_of` -- дата отсечки.

Формат ответа:

- `predictions` -- массив точек (date, value, lower, upper).
- `model_name` -- использованная модель.
- `confidence` -- оценка достоверности.

## 3. Анализ подходов к моделированию

### 3.1 Временные ряды (классические)

**ARIMA / SARIMA.**
Подходят для данных с выраженной сезонностью (недельные циклы коммитов). Преимущества: простота, интерпретируемость, низкие требования к данным. Недостатки: линейность, слабая работа с множественными рядами.

**Prophet (Meta).**
Хорошо обрабатывает праздники, пропуски данных. Подходит как baseline-модель для прогнозирования недельной активности разработчиков.

### 3.2 Нейросетевые архитектуры

**LSTM (Long Short-Term Memory).**
Рекуррентная сеть, способная улавливать долгосрочные зависимости в последовательностях. Хорошо подходит для прогнозирования активности с учетом контекста (отпуска, спринты). Требует достаточного объема данных (от 6 месяцев истории на пользователя). Обучение стандартно на PyTorch/TensorFlow.

**Samba (State Space Model + Mamba).**
Гибридная архитектура, сочетающая пространство состояний (SSM) и механизм Mamba для эффективного моделирования длинных последовательностей. Преимущества: линейная сложность по длине последовательности, подходит для обработки данных множества пользователей одновременно. Перспективная модель для масштабирования на тысячи разработчиков.

**KAN (Kolmogorov-Arnold Networks).**
Альтернативная архитектура, заменяющая линейные слои обучаемыми сплайнами на ребрах графа. Преимущества: высокая точность на малых объемах данных, интерпретируемость. Подходит для сценариев, где данных мало (новые сотрудники, небольшие команды).

### 3.3 Рекомендация

Предлагается трехэтапный подход:

1. **Baseline (месяцы 1-3):** Prophet для каждого пользователя. Простое развертывание, быстрая валидация.
2. **Продвинутая модель (месяцы 3-6):** LSTM для пользователей с достаточной историей. A/B тестирование против Prophet.
3. **Масштабирование (месяцы 6-12):** Samba для обработки всех пользователей единой моделью. KAN как fallback для пользователей с малым объемом данных.

## 4. Архитектура ML-сервиса

### 4.1 Компоненты

- **Model Serving (FastAPI / TorchServe)** -- HTTP-сервер, обслуживающий запросы прогнозирования.
- **Feature Store** -- хранилище предобработанных признаков, извлеченных из `git_activities`.
- **Training Pipeline** -- периодическое переобучение моделей (Airflow / Prefect).
- **Model Registry (MLflow)** -- версионирование моделей и метрик.

### 4.2 Интеграция с GConsus

```
GConsus Backend
  |
  |-- POST /predict  -->  ML Service (FastAPI)
  |                            |
  |                            |-- Feature Store (Redis / PostgreSQL view)
  |                            |-- Model Registry (MLflow)
  |
  |<-- PredictionResponse
```

ML-сервис читает признаки из Feature Store и возвращает прогноз с доверительными интервалами. Backend кеширует результат и отдает фронтенду.

### 4.3 Признаки (Features)

Для каждого пользователя формируется временной ряд:

- Дневные агрегаты: commits_count, prs_count, reviews_count, issues_count, lines_added, lines_deleted.
- Производные: rolling_7d_mean, rolling_30d_mean, day_of_week, is_holiday.
- Контекстные: team_size, active_repos_count, sprint_phase (при интеграции с Jira).

## 5. Дорожная карта

### Этап 1: Подготовка (месяцы 1-2)

- Создать Feature Store -- представление (materialized view) над `git_activities` с дневными агрегатами.
- Развернуть FastAPI сервис с Prophet-моделью.
- Подключить ML коннектор в бэкенде (`ML_ENABLED=true`).
- Добавить эндпоинт `/analytics/users/{id}/forecast` в API.
- Добавить визуализацию прогноза на фронтенде (линия + доверительный интервал).

### Этап 2: Улучшение модели (месяцы 3-5)

- Реализовать LSTM-модель на PyTorch.
- Настроить Training Pipeline (еженедельное переобучение).
- Интегрировать MLflow для трекинга экспериментов.
- A/B тестирование Prophet vs LSTM.
- Добавить прогнозы на уровне команд.

### Этап 3: Масштабирование (месяцы 6-9)

- Реализовать Samba-модель для обработки множества пользователей.
- KAN-модель для новых пользователей с малым объемом данных.
- Обнаружение аномалий: резкое падение активности, нетипичные паттерны.
- Уведомления менеджерам при отклонении от прогноза.

### Этап 4: Корпоративная интеграция (месяцы 9-12)

- Интеграция с Jira/YouTrack для контекстных признаков (спринты, задачи).
- Интеграция с HR-системой для учета отпусков и больничных.
- Интеграция с CI/CD для метрик сборок и деплоев.
- Дашборд руководителя: сводный прогноз по отделу.
- Экспорт отчетов в PDF.

## 6. Требования к инфраструктуре

### 6.1 Compute

- Training: GPU-инстанс (NVIDIA T4 или выше), 16 GB RAM.
- Serving: CPU-инстанс, 4 GB RAM (Prophet/KAN), 8 GB RAM (LSTM/Samba).

### 6.2 Storage

- Feature Store: 10-50 GB (зависит от количества пользователей и глубины истории).
- Model Registry: 1-5 GB (версии моделей).

### 6.3 Scheduling

- Переобучение моделей: раз в неделю (Airflow DAG или cron job).
- Обновление Feature Store: раз в день после завершения синхронизации.

## 7. Метрики успеха

- MAPE (Mean Absolute Percentage Error) прогноза коммитов менее 20% на горизонте 7 дней.
- Точность обнаружения аномалий (precision/recall) не менее 80%.
- Время ответа ML-сервиса менее 500ms (p95).
- Покрытие: прогнозы доступны для 90% активных пользователей.

## 8. Риски и ограничения

- Малый объем данных на начальном этапе (менее 6 месяцев) ограничивает точность LSTM.
- Сезонность (праздники, отпуска) требует внешних данных для корректного учета.
- Этические аспекты: прогнозы не должны использоваться для давления на сотрудников. Система предназначена для планирования и выявления системных проблем, а не для оценки индивидуальной производительности.
